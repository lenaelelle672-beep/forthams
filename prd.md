<channel|># PRD：ForthAMS 多租户隔离强制校验机制

## 项目概述

在 ForthAMS 安全红线验证中，多租户数据隔离是核心安全底线。当前系统存在跨租户数据越权风险隐患。本需求旨在通过拦截层（Interceptor/Filter）实现 TenantID 的强制上下文绑定与查询过滤，确保任何数据库操作必须经过租户身份校验，严禁裸 SQL 拼接，构建强阻断的安全防线。

## 核心目标

1.  **建立不可绕过的租户上下文机制**：从 JWT 或安全上下文中提取并锁定 TenantID
2.  **实现查询层强制隔离**：所有数据访问必须携带有效的租户标识
3.  **防止渗透攻击**：禁止任何未经验证的跨租户数据读取

## 功能需求列表

| 编号 | 功能项 | 描述 | 优先级 |
| :--- | :--- | :--- | :--- |
| F1 | TenantContext 构建 | 基于 `JwtUtil.java` 解析 Token 并将 tenant_id 注入 ThreadLocal 安全上下文，确保请求链路中租户信息始终可用且不可篡改。 | P0 |
| F2 | AssetController 强制拦截 | 在 Controller 层通过 AOP 或 Filter 校验当前操作的 TenantID 是否与资源所属租户一致，不匹配直接抛出 AccessDeniedException。 | P0 |
| F3 | 多租户数据库隔离模拟查询 | 实现基于 tenant_id 的动态 SQL 条件注入（使用 MyBatis/JPA 参数绑定），禁止任何裸拼接 SQL 绕过过滤。 | P0 |
| F4 | TenantID 安全上下文回退保护 | 当 Context 中缺失有效租户标识时，默认拒绝所有数据访问请求并记录安全告警日志。 | P1 |
| F5 | 操作审计日志增强 | 在拦截层记录每一次跨租户尝试的攻击行为，包括来源 IP、操作类型及被阻断的具体资源 ID。 | P2 |

## 非功能需求

### 性能要求
- **低延迟**：租户上下文提取与校验逻辑增加的响应耗时应控制在 <5ms 内
- **无状态扩展性**：ThreadLocal 设计需支持请求结束后自动清理，避免内存泄漏并兼容并发场景

### 安全要求
- **强阻断机制**：任何未携带有效 TenantID 的查询必须默认返回空结果或抛出异常，严禁降级为全量查询
- **防注入设计**：数据库过滤条件必须通过参数化绑定（PreparedStatement），禁止字符串拼接构建 WHERE 子句

### 可维护性要求
- **代码位置规范**：逻辑集中在 `JwtAuthenticationFilter` 或新建的隔离切面中，避免散落在业务逻辑层
- **可观测性**：拦截失败需输出结构化日志，便于安全审计与异常溯源

## 验收标准概览

| 测试项 | 预期结果 | 通过标准 |
| :--- | :--- | :--- |
| 有效 Token 请求 | 返回租户内数据 | ✅ TenantID 与资源所属一致时正常访问 |
| 无有效 Token 请求 | 直接拒绝请求 | ✅ 被拦截，返回 401/403 |
| Token 所属租户 A 查询租户 B 数据 | 结果为空或抛异常 | ✅ 跨租户查询被阻断，不泄露任何数据 |
| SQL 层注入测试 | 无越权读取行为 | ✅ 参数化绑定生效，无法通过修改参数绕过隔离 |
| 并发请求下的上下文一致性 | 各线程 TenantID 正确匹配 | ✅ ThreadLocal 清理机制正常工作 |
