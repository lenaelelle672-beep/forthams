**生成时间**: 2026-04-01

## 核心问题 (Core Challenge)

基于Audit结果，我们面临的核心问题:

**如何在无人值守模式下，高效补全80%缺失功能 + 实现100%测试覆盖？**

---

## 方案对比 (Solution Comparison)

### 方案A: 串行开发模式 (Sequential Development)

**思路**: 先完成所有后端模块 → 再连接前端 → 最后写测试

#### 优点 ✅
- 逻辑清晰，易于跟踪进度
- 后端API完整后前端集成更顺畅
- 适合单人开发

#### 缺点 ❌
- **开发周期长** (预计10-12周)
- 测试延后，bug发现晚
- 前端集成延迟，无法早期验证
- 不符合"无人值守"要求 (需要大量人工决策)

#### 风险
- 🔴 **High**: 集成阶段发现架构问题导致返工
- 🔴 **High**: 测试覆盖不足，质量无保障

#### 适用场景
- 有充足时间
- 需要深度人工审查
- 不适合本次需求 ❌

---

### 方案B: 测试驱动开发模式 (TDD Approach)

**思路**: 先写测试 → 实现功能使测试通过 → 前端集成

#### 优点 ✅
- 测试覆盖率100%保障
- bug发现早，修复成本低
- 代码质量高

#### 缺点 ❌
- **开发速度慢** (测试编写时间长)
- 前期投入大
- 需要高水平TDD技能
- 不适合快速补全80%功能的场景

#### 风险
- 🟠 **Medium**: 过度设计导致进度延误
- 🟡 **Low**: 测试维护成本高

#### 适用场景
- 新项目从零开始
- 质量要求极高
- 不太适合本次需求 ⚠️

---

### 方案C: 分层并行开发模式 (Layered Parallel Development) ⭐ **推荐**

**思路**: 按优先级分批次，每批次并行开发"后端+前端+测试"

#### 核心策略

**批次划分**:
1. **Batch 1** (高优先级基础模块) - 1周
   - 后端: AssetCategory, User, Dept, Role CRUD
   - 前端: API集成 + 按钮Handler实现
   - 测试: 单元测试 + 组件测试

2. **Batch 2** (核心业务模块) - 2周
   - 后端: Maintenance, Inventory, IdleAsset
   - 前端: 表单验证 + 错误处理
   - 测试: 集成测试 + E2E关键流程

3. **Batch 3** (高级功能模块) - 1周
   - 后端: Compensation, Approval, Attachment
   - 前端: 文件上传 + 高级搜索
   - 测试: 完整E2E覆盖

4. **Batch 4** (测试完善与文档) - 1周
   - 补全所有模块测试
   - 生成测试文档
   - 修复已知bug

#### 优点 ✅
- **开发速度快** (并行开发，5周完成)
- 每批次都有可交付成果
- 测试同步进行，质量有保障
- 前端集成早期验证，避免返工
- **适合无人值守模式** (每批次独立，自动化程度高)

#### 缺点 ❌
- 需要良好的任务拆分能力
- 批次间可能需要接口调整

#### 风险
- 🟡 **Low**: 批次间依赖导致阻塞 (通过合理规划避免)
- 🟢 **Very Low**: 代码质量问题 (测试同步覆盖)

#### 为什么最适合本次需求？

1. **符合无人值守**: 每个批次都是自包含的，可自动化执行
2. **快速交付**: 5周完成vs其他方案10-12周
3. **质量保障**: 测试同步开发，覆盖率100%
4. **风险可控**: 分批次暴露问题，及时修正

---

### 方案D: AI辅助自动生成模式 (AI-Assisted Generation)

**思路**: 使用AI批量生成CRUD代码 → 人工审查 → 测试

#### 优点 ✅
- **极快的开发速度** (2-3周)
- 代码风格一致
- 适合重复性CRUD

#### 缺点 ❌
- **生成代码质量参差不齐**
- 复杂业务逻辑难以生成
- 需要大量人工审查和修改
- 测试生成质量低

#### 风险
- 🔴 **High**: 生成代码的隐藏bug
- 🔴 **High**: 业务逻辑错误

#### 适用场景
- 辅助工具，不作为主要方案
- 可用于生成重复性代码（如DTOs、Mappers）

## 测试策略选型 (Testing Strategy Selection)

### 选项1: 最小化测试策略 (Minimal Testing)

- 仅核心业务路径测试
- 测试覆盖率目标: 60%
- **不推荐** ❌ (不符合用户要求"专业的单元测试")

### 选项2: 全覆盖测试策略 (Comprehensive Testing) ⭐ **推荐**

#### 后端测试策略

1. **单元测试** (JUnit 5 + Mockito)
   - 目标: 100%覆盖Service层
   - 模式: 每个业务方法1个测试
   - 工具: MockMvc (Controller层)

2. **集成测试** (Spring Boot Test)
   - 目标: 覆盖主要业务流程
   - 模式: 端到端API测试
   - 工具: TestRestTemplate + H2内存数据库

3. **数据库测试**
   - 模式: Mapper层测试
   - 工具: MyBatis-Plus测试支持

#### 前端测试策略

1. **组件测试** (Vitest + React Testing Library)
   - 目标: 100%覆盖UI组件
   - 模式: 按钮点击、表单提交、搜索框输入
   - 示例:
     ```typescript
     test('保存按钮点击后调用API', async () => {
       const { getByRole } = render(<AssetForm />);
       const saveButton = getByRole('button', { name: /保存/i });
       await userEvent.click(saveButton);
       expect(mockApiService.create).toHaveBeenCalled();
     });
     ```

2. **E2E测试** (Playwright)
   - 目标: 覆盖关键业务流程
   - 场景:
     - 用户登录 → 创建资产 → 查询 → 编辑 → 删除
     - 创建盘点任务 → RFID扫描 → 差异处理
     - 提交审批 → 审批通过 → 查看历史
   - 示例:
     ```typescript
     test('完整资产管理流程', async ({ page }) => {
       await page.goto('/login');
       await page.fill('[name="username"]', 'admin');
       await page.fill('[name="password"]', 'admin123');
       await page.click('button:has-text("登录")');
       await page.click('text=资产台账');
       await page.click('button:has-text("新建资产")');
       await page.fill('[name="assetNo"]', 'TEST-001');
       await page.click('button:has-text("保存")');
       await expect(page.locator('text=创建成功')).toBeVisible();
     });
     ```

3. **快照测试** (可选)
   - 确保UI不会意外变化


## 实施原则 (Implementation Principles)

### 1. 测试先行 (Test-First for Critical Paths)

- 核心业务逻辑先写测试
- 简单CRUD可以先实现后测试
- 确保每个功能点都有对应测试

### 2. 小步快跑 (Incremental Delivery)

- 每个批次完成后立即验证
- 不等所有功能完成再测试
- 发现问题立即修复

### 3. 文档同步 (Documentation in Sync)

- 测试完成后立即生成文档
- 文档包含:
  - 功能清单 (带checkbox)
  - 测试用例列表
  - 测试结果汇总
  - 已知问题清单

### 4. 质量优先 (Quality First)

- 不为速度牺牲质量
- 测试覆盖率硬性要求
- 代码审查 (LSP错误修复)

## 风险缓解计划 (Risk Mitigation)

### 风险1: 批次间依赖导致阻塞

**缓解措施**:
- 提前设计接口契约
- 使用Mock数据避免依赖
- 批次顺序合理安排

### 风险2: 测试编写时间超预期

**缓解措施**:
- 使用测试模板减少重复工作
- AI辅助生成测试框架
- 聚焦核心业务路径

### 风险3: 前端API集成问题

**缓解措施**:
- 早期验证API契约
- 使用MSW隔离测试
- 集成测试覆盖全流程
