Step 1

面向一个项目，应该如何拆解

面对一个项目或软件需求，不能直接把一句话丢给 AI。正确方式是先把项目拆成层级，让 AI 知道目标、上下文、边界和验收。

业务目标层
  用户要解决什么问题
  成功标准是什么

产品需求层
  角色是谁
  业务流程是什么
  边界条件是什么
  异常情况是什么

技术架构层
  后端是什么语言
  前端是什么框架
  数据库怎么组织
  API 怎么暴露
  测试框架是什么

代码实现层
  哪些 Controller / API 要改
  哪些 Service / Domain 要改
  哪些 DTO / Entity / Mapper 要改
  哪些测试要补

验收层
  编译是否通过
  单元测试是否通过
  API 行为是否符合预期
  是否没有破坏旧功能
业务

先讲目标

不要直接进入代码。先明确用户为什么要这个功能，做到什么程度算成功。

边界

再讲限制

哪些文件可以改，哪些文件不能改，哪些行为不能发生。

路径

定位入口

真实项目必须知道后端根目录、前端根目录、Controller、Service、DTO、Entity 等位置。

验收

最后讲完成

不是“看起来写完了”，而是编译、测试、AC 验收都要能过。

Step 2

如何与 AI 交流

AI 最怕上下文缺失。好的输入不是更长，而是更结构化。

不推荐

一句话愿望

帮我实现资产报废流程。

问题：模型只能猜项目结构、业务边界、文件路径和完成标准。

推荐

结构化任务输入

当前项目：Java Spring Boot + 前端管理系统
业务目标：实现资产报废申请与审批
涉及角色：申请人、审批人、管理员
可能文件：Controller / Service / DTO / Entity / Mapper
禁止修改：环境配置、锁文件、无关模块
验收标准：mvn compile 通过，核心 API 行为符合 AC
失败处理：只回滚本轮 touched files，保留用户已有改动
讲解重点：prompt 不是魔法咒语，而是工程合同。GSD 做的事情，就是把这种结构化沟通流程自动化。

Where it fits

GSD Harness 可以用来做什么

它不是 IDE 补全工具，而是适合项目级、多文件、可验收的自动开发任务。

适合
  • 真实项目里的多文件功能开发。
  • Java / Spring Boot 后端功能补齐。
  • Controller / Service / DTO / Mapper 级别改造。
  • TDD 风格需求实现。
  • 编译、测试、验收闭环。
  • 长任务自动化开发。
  • 项目级经验沉淀。
  • 后续手机端观察和人工干预。
不适合替代
  • IDE 里的毫秒级代码补全。
  • 没有验收标准的开放式头脑风暴。
  • 完全无项目上下文的随手问答。
  • 没有工程边界约束的“随便改改”。
Cursor / Copilot 更像副驾驶。GSD 更像带施工日志、验收标准、回滚机制和长期记忆的自动施工队。
Terms

先解释两个核心词：Harness 和 GSD

听众理解这两个词后，再看后面的架构图会更自然。

Harness

运行框架 / 约束装置

Harness 原意是“马具 / 约束装置”。在软件工程里，Harness 常指一个运行框架，它把被执行对象包起来，提供环境、输入、执行控制、观测能力、结果收集和错误处理。

环境输入执行控制观测结果收集错误处理
GSD

项目级自动开发流程引擎

GSD 是 Get Shit Done 的缩写。在这里它不是单个 Agent，而是负责接需求、生成规格、定位代码、生成变更契约、拆解执行策略、调用执行器、跑测试验收、记录状态和沉淀记忆的流程引擎。

需求规格定位执行验收记忆

GSD

内部项目主流程

负责项目级计划、执行策略、状态、记忆和验收，是当前已经开发的核心。

Hermes / Claude

未来监督者

只监督 GSD 运行情况，不直接改项目代码，避免多个执行器写盘协议分裂。

手机端

未来控制台

观察任务阶段、worker 状态、evaluator score，并支持暂停或调整。

Lifecycle

这条链路是讲架构时最重要的主线。

1

自然语言输入，例如“实现资产报废流程”。

2

3

Control Plane vs Execution Plane

控制平面和执行平面

这是 GSD 当前设计中最关键的边界：控制平面决定“应该怎么做”，执行平面负责“真正改代码”。

控制平面

负责计划和约束

  • GsdOrchestrator
  • GsdPlanner
  • GsdLocalizer
  • ContractNegotiator
  • BuilderPlanRunner
  • GateEngine
  • ExecutionPolicyEngine
  • MutationContract
  • PlanState / WAL / Memory
执行平面

负责真实写代码

  • FreecodeExecutor 是主路径唯一代码执行器。
  • 通过 XML tool loop 读文件、替换、写文件、运行命令。
  • 强制 pre-read，减少幻觉修改。
  • 失败时按 attempt touched files 恢复，不做全仓库危险回滚。
Modules

按架构图拆模块

这里可以作为网页介绍的主体部分。每个模块都说明职责、解决的问题和边界。

GsdOrchestrator：主流程编排

接收需求，启动 sprint / epoch，调用 Planner、Localizer、Contract、TestWriter、BuilderPlanRunner、FreecodeExecutor 和 Evaluator，并管理 checkpoint、WAL、metrics、长期记忆。它是流程总线，不直接写业务代码。

GsdPlanner：生成规格 / PRD

把用户自然语言需求转成 spec.md，明确需求背景、边界、验收基准和开发切入层级。未来可以增强为结构化 PRD、user story、task DAG。

GsdLocalizer：代码定位

定位可能受影响的文件，提供类、方法、行级线索。它解决的问题是：不要让模型自己在项目里瞎找。

ContractNegotiator：验收契约

把需求变成验收条件 AC，让 Evaluator 可以判断任务是否完成。

BuilderPlanRunner：执行前计划

提取目标文件、归一化路径、生成 MutationContract、调用 GateEngine、调用 ExecutionPolicyEngine、预加载文件内容，并保存 builder plan artifact，例如 .gsd/plans/<task_id>/iteration-<n>-builder-plan.json。

GateEngine：安全门

检查工作区是否 dirty，在 worker mode 下跳过主进程 dirty gate，并为执行策略提供风险信号。核心价值是避免自动化系统覆盖用户已有改动。

ExecutionPolicyEngine：执行策略

决定 force_solo、auto、force_team。判断依据包括文件数量、是否涉及基础共享文件、workspace 是否 dirty、是否包含测试文件、是否存在可并行分组。

MutationContract：变更边界

明确允许写哪些文件，阻止高风险文件自动改写，提供路径别名归一。这是防止模型漂移的重要机制。

FreecodeExecutor：唯一代码执行器

负责 XML 工具协议、read / replace / write / run_cmd、强制 pre-read、MINIONS 双推理、attempt 级回滚、team mode worker 执行。它是当前唯一主路径代码写入者。

Freecode Team Mode：内部任务并发

将复杂任务拆成 TaskNode，wave 1 写测试，wave 2 实现，文件不冲突则并发，worker 完成后打标。

SwarmSupervisor：外层多角色分工

根据 [SWARM] 指令启用外层角色分工，每个角色用 SwarmRoleProxy，每个角色内部仍然用 FreecodeExecutor。注意：Swarm 是外层并发，Freecode Team 是内层并发，两层不能无限套娃。

TaskStatusReporter：后台状态线程

接收 started / done / failed 事件，写 PlanState 和 WAL。它不调用模型，不消耗 token。

记忆层：ContextOptimizer / RepoMap / Graphify / LanceDB

ContextOptimizer 负责短期上下文压缩，RepositoryMapMemory 负责项目 AST 图，GraphifySkill 负责外部资料图谱，LanceDBMemory 负责长期语义记忆。

GsdEvaluator：质量门

负责编译、测试、AC 验收和失败反馈。它决定任务是否真正完成，而不是只看代码有没有生成。

Mechanisms

功能机制细化

这些是 GSD 能够从 demo 走向真实项目的关键机制。

XML Tool Loop

为什么不用普通 Patch

旧方案 Opencode Patch 格式复杂，模型容易输出错。XML 工具协议让模型先 read，再 replace / write，每次只做一步，失败可以纠正。

<read_file path="backend/src/main/java/..." />
<replace file="backend/src/main/java/...">
  <old>真实旧代码</old>
  <new>新代码</new>
</replace>
双推理

本地读，云端写，执行器守门

只用云端质量高但成本高、慢、容易 429；只用本地成本低但格式和质量不稳。新的双推理让本地模型先低成本探索和草拟，云端模型主导关键写动作，Executor Guard 负责硬约束。

local scout draft
  -> cloud final action
  -> executor guard
  -> execute
Attempt Rollback

只恢复本轮 touched files

编译失败时不再全仓库清场，避免误伤用户已有改动。恢复的是本轮执行前记录的 preimage。

compile failed
  -> restore touched files only
  -> inject feedback
  -> next attempt
WAL + PlanState

怎么知道做到哪了

Checkpoint 记录粗粒度恢复点，WAL 记录事件历史，PlanState 记录当前 step / worker 状态，TaskStatusReporter 后台统一打标。

FilePartition

怎么避免并发冲突

同一批并发 worker 不能写同一个文件，未知边界任务自动串行。worktree / merge 只是兜底，不是第一防线。

Vaccine Memory

怎么避免重复踩坑

连续失败后由 Investigator 总结死路，写入 LanceDB vaccine，下次相似需求召回。

Agent Design Evolution

从社区思想到 GSD 的执行方案

这一节回答两个问题：为什么要双推理？为什么不是简单照搬 MINIONS、Aider、SWE-agent 或 AutoGen？

以前

云端 verify 可以覆盖本地动作

早期实现是本地模型先出草稿，云端模型再 verify / rewrite，最后直接执行云端输出。

local draft
  -> cloud verify / rewrite
  -> execute cloud output

这个方案能提高质量，但有一个严重问题：云端可能把已经准备好的写动作改回 read_file / grep / ls，导致模型在项目里反复读、迟迟不写。

现在

云端主导写，执行器守门

新的方案不是简单“本地为准”或“云端为准”，而是按动作风险分工：本地负责便宜探索，云端负责关键写动作，执行器负责硬约束。

local scout
  -> cloud finalizer
  -> executor guard
  -> XML tool execution

云端可以改进同目标写动作；但不能无理由把写阶段降级成只读循环，也不能越过 MutationContract 写其它文件。

参考思想核心启发GSD 怎么吸收
MINIONS / MinionS本地小模型处理大量上下文，云端大模型处理聚焦后的关键问题，降低成本和 429。本地模型作为 Scout，先读上下文、压缩信息、提出 draft；云端模型只在写动作、失败修复和高风险动作上介入。
Aider Architect Mode把架构推理和具体编辑分开，强模型负责更高质量的判断。云端模型作为 Cloud Finalizer，负责最终 replace / write_file 动作；本地模型不再独自承担高质量代码修改。
SWE-agent / mini-SWE-agent每一步都是可审计 action，工具执行结果进入下一轮，而不是让模型自由发挥。FreecodeExecutor 使用 XML tool loop：read_file、replace、write_file、run_cmd、done，每一步都有结果、有状态、有失败反馈。
OpenHands / AutoGen多角色协作需要 selector、termination 和 guard，否则容易无限对话或抢执行权。GSD 保留子代理和 Swarm，但所有写代码动作都收口到 FreecodeExecutor，并由 MutationContract、FilePartition、PlanState、WAL 约束。
事务 / 编译器思想不要让智能体直接改世界；先生成 mutation proposal，再验证、执行、记录、必要时恢复。Executor Guard 检查路径、语法、replace 精确匹配、危险命令、状态机和 attempt 级 preimage rollback。
Local Scout

本地模型做什么

  • 低成本读取上下文。
  • 总结已知信息和不确定点。
  • 提出候选修改点。
  • 生成初版 XML action 或结构化 draft。
  • 减少云端需要处理的 token 和请求次数。
Cloud Finalizer

云端模型做什么

  • 基于本地 draft 做高质量推理。
  • 生成最终写动作。
  • 修正本地模型的代码质量问题。
  • 在测试失败后生成更可靠的修复。
  • 必要时 request_read，但必须被状态机允许。
Executor Guard

执行器做什么

  • 检查 XML 是否合法。
  • 检查是否越权写文件。
  • 阻止 read loop。
  • 检查语法和 replace old block。
  • 记录 WAL、PlanState 和 touched files，失败时只恢复本轮文件。

动作风险决定谁主导

GSD 的关键不是固定“谁说了算”，而是按动作风险切换主导权。

动作类型主导者原因
ls / grep / read_outline / 首次 read_file本地模型便宜、低风险、可降低云端请求。
replace / write_file云端模型代码质量要求更高，本地模型容易写错。
核心 service / entity / controller 修改云端 + Guard高风险，需要强模型推理和硬约束校验。
连续只读后继续 readExecutor Guard直接阻止，防止 read loop。
测试失败后的修复云端模型需要理解失败输出和代码关系。
 
if action is read:
  local-first
  guard checks loop state

if action is write:
  cloud-final
  guard checks scope / syntax / state

if cloud fails or 429:
  use local draft only when guard passes

if local and cloud disagree:
  prefer cloud for writes
  prefer guard against read loops
Context & Memory

上下文与记忆层

GSD 的记忆层不是为了“把所有东西都塞给模型”，而是在正确的时间，把正确的上下文放进模型窗口。

层级模块解决的问题
短期记忆ContextOptimizer单次任务上下文膨胀、模型遗忘最初目标。
项目结构记忆RepositoryMapMemory / GraphifySkill模型不知道项目结构、入口、类和方法位置。
长期经验记忆LanceDBMemory历史踩坑、成功经验、失败疫苗无法复用。
Comparison

和主流 IDE / Code Agent 的差异

这部分用于回答“为什么不用 Cursor / Copilot / Claude Code 直接做”。

系统上下文策略优点局限
Cursor项目索引、打开文件、当前光标、embedding 检索、后台压缩IDE 体验好，响应快任务级 WAL / 可恢复状态较弱
GitHub Copilot当前文件、邻近上下文、IDE 语义信息补全强，延迟低项目级规划和验收弱
GPT Coding当前对话、当前文件/选区、工具结果、系统指令预算泛化强长任务状态需要外部系统承接
Claude Codeagentic tool loop、read/search/edit、长上下文、压缩长任务能力强持久化状态和项目级验收需要外部机制
GSDRepoMap + Graphify + LanceDB + WAL + PlanState + ContextOptimizer项目级流程、可恢复、可审计架构复杂，执行成本更高
Status

当前已解决的问题与后续计划

这部分适合放在结尾，让听众知道当前能力边界。

已解决
  • 主路径统一到 FreecodeExecutor。
  • VibeExecutor 退为 legacy。
  • MutationContract 控制写入边界。
  • ExecutionPolicy 控制 solo / team。
  • 本地读、云端写、执行器守门的双推理方向已确定。
  • Graphify 调用链修复。
  • Java RepoMap 支持。
  • LanceDB 项目内长期记忆。
  • 后台状态线程。
  • WAL step 级事件。
  • Freecode Team worker 状态打标。
  • repo 级危险回滚移除。
还没做
  • Hermes / Claude 上帝监督者还没开发。
  • 手机端观察和调整还没开发。
  • 结构化 PRD / user story / task DAG 还没完全上移到控制平面。
  • Swarm role worker 状态还可以进一步统一接入 reporter。
  • Evaluator 对 Java compile failure 还可以更强硬。
Glossary

术语速查

用于现场解释听众不熟悉的工程词。

WAL

Write-Ahead Log，事件先写日志，方便恢复和审计。

AC

Acceptance Criteria，验收条件。

429

Too Many Requests，云端模型或 API 调用过频时常见。

Context Window

模型一次能看到的上下文窗口，不是无限记忆。

MutationContract

变更契约，规定哪些文件允许改、哪些禁止改。

Vaccine Memory

失败疫苗，把连续失败的根因写入长期记忆。
